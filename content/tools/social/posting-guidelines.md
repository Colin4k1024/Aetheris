# Posting Guidelines

Best practices and guidelines for Aetheris X (Twitter) content.

## Brand Voice

### We Are
- **Technical but accessible** — Complex topics, explained simply
- **Confident but humble** — Know our strengths, acknowledge limitations
- **Helpful and educational** — We share knowledge generously
- **Community-focused** — We celebrate contributors and users
- **Transparent** — Share both wins and challenges

### We Are Not
- Salesy or hype-driven
- Jargony without explanation
- dismissive of alternatives
- Passive aggressive
- Clickbait

## Content Pillars

1. **Technical Education** — How-tos, deep dives, architecture explainers
2. **Product Updates** — Releases, features, improvements
3. **Community** — Contributors, users, use cases
4. **Industry Perspective** — Agent patterns, durability, production AI

## Posting Frequency

| Content Type | Frequency | Notes |
|--------------|-----------|-------|
| Single tweet | 3-5/week | Tips, quick updates |

## Optimal Posting Times (PST/PDT)

- **Best:** Tuesday-Thursday, 9am-11am
- **Good:** Monday-Friday, 6am-8am, 5pm-7pm
- **Avoid:** Friday 4pm+, weekends (unless engagement post)

## Thread Guidelines

### Thread Anatomy

1. **Hook Tweet (Tweet 1)**
   - Grab attention immediately
   - Use numbers, bold claims, or questions
   - End with "🧵" to signal a thread
   - Example: "We rewrote our scheduler 3 times. Here's what we learned 🧵"

2. **Context Tweets (Tweets 2-3)**
   - Set up the problem or topic
   - Who is this for? What's the context?

3. **Core Content (Tweets 4-7)**
   - The actual value
   - Use code snippets, diagrams, or examples
   - One key point per tweet

4. **Closing Tweet (Tweet 8+)**
   - Summary or key takeaway
   - Call to action (try it, reply, share)
   - Link to docs or repo

### Thread Best Practices

- Keep tweets under 260 characters (leave room for retweets with comment)
- Use numbering: "1/" "2/" etc. for easy reading
- Add value in every tweet — don't just say "next tweet"
- End with a question or invitation to reply
- Engage with everyone who replies

## Engagement Guidelines

### Responding to Replies
- Reply to all comments within 24 hours
- Be helpful and genuine
- Ask follow-up questions
- Thank contributors and positive commenters
- For critical feedback: acknowledge, don't defensive

### Quote Tweets
- Add value, don't just amplify
- "This is great because..." or "Building something similar..."
- Engage with the quoted content

### Retweets
- Always add a comment, don't just RT
- Amplify community wins
- Share relevant third-party content occasionally

## Tone Examples

### Announcement (Confident, exciting)
> 🚨 Aetheris 2.0 is here.
> 
> Crash recovery that actually works. At-most-once tool execution. Deterministic replay.
> 
> The production-ready agent runtime you've been waiting for.
> 
> [link]

### Educational (Helpful, patient)
> Struggling with agent reliability?
> 
> Here's the problem: most agents are "fire and forget."
> 
> They crash, they lose state, they start over.
> 
> Aetheris changes that. Let me explain 🧵

### Community (Grateful, warm)
> 🎉 Big thanks to @username for contributing the new checkpoint API!
> 
> Your PR saved developers hours of debugging.
> 
> This is what open source community looks like ❤️

### Technical (Precise, thorough)
> Hot take on agent idempotency:
> 
> It's not about "making calls once."
> It's about designing for the case where your call IS executed multiple times.
> 
> At-most-once ≠ Idempotent
> 
> You need both.

## Crisis Management

### If You Make a Mistake
1. Acknowledge quickly
2. Correct factually
3. Apologize genuinely if needed
4. Move forward

### If Someone Critiques
1. Listen fully
2. Don't be defensive
3. Thank them if feedback is valid
4. Address technical concerns technically

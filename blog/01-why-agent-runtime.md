# 为什么需要 Agent Runtime？——从脆弱的脚本到可靠的进程

> 当 AI Agent 从玩具变成生产系统，一场执行层的架构革命正在发生。

## 0. 一个真实的崩溃故事

你写了一个 AI Agent，用于自动处理客户退款请求。流程很简单：

1. 读取用户退款申请
2. 调用风控 API 检查风险
3. 调用支付 API 执行退款
4. 发送确认邮件

某个深夜，Agent 正在处理第 1000 单退款时，服务器突然重启了。

重启后，你发现：
- **第 999 单**：支付 API 已经调用成功，但还没来得及发邮件 —— **用户收到了钱，但没收到确认邮件**
- **第 1000 单**：完全没执行 —— **用户既没收到钱，也没收到任何通知**

你artner 只能手动查日志、对账单、平复客户愤怒。

这不是假设。这是每个想把 Agent 投入生产的人都会遇到的真实噩梦。

## 1. 问题的本质：Agent 是「进程」还是「脚本」？

传统 AI 框架把 Agent 当**脚本**看待：

```
用户请求 → 调用 LLM → 返回结果 → 结束
```

这对应的是**无状态请求-响应**模式，适合聊天，但不适合真正的**长时任务**。

真实的 Agent 工作流是这样的：

```
┌─────────────────────────────────────────────────────────────┐
│  退款 Agent                                                  │
│  ├─ 读取申请 (30s)                                          │
│  ├─ 风控检查 (API 调用，2s)                                  │
│  ├─ 退款执行 (支付 API，5s)    ← 外部副作用！               │
│  ├─ 发送邮件 (SMTP，3s)       ← 外部副作用！               │
│  └─ 更新数据库 (1s)            ← 外部副作用！               │
└─────────────────────────────────────────────────────────────┘
              ⬇️ 可能会发生的一切
              - 服务器崩溃
              - 网络超时
              - API 限流
              - 人工审批卡住 3 天
```

问题来了：**当崩溃发生时，哪些步骤已经执行了？哪些还没执行？能否安全地恢复？**

传统框架的答案是：**不知道。** 要么全部重来（可能重复扣款），要么手动处理（费时费力）。

## 2. 为什么 Agent 需要「操作系统」？

让我们换个角度思考。

### 2.1 进程 vs 脚本

| 维度 | 脚本 | 进程 |
|------|------|------|
| 生命周期 | 单次运行 | 持久存在 |
| 状态 | 无状态 | 有状态 |
| 崩溃恢复 | 重头来过 | 从检查点恢复 |
| 外部副作用 | 不保证 | At-most-once |
| 暂停/恢复 | 不支持 | 支持 |

**Agent 本质上就是一个「AI 进程」**—— 它有状态、要执行多步操作、有外部副作用、需要持久运行。

而我们熟悉的操作系统（Linux、Kubernetes）就是为「进程」设计的：
- 进程崩溃了？systemd/Kubernetes 会重启它
- 进程需要恢复状态？检查点（checkpoint）机制
- 进程之间需要协作？进程间通信（IPC）

**那谁来管理 AI Agent 这个「进程」？**

答案就是：**Agent Runtime**。

## 3. Agent Runtime 是什么？

Agent Runtime 是介于 Agent 逻辑和基础设施之间的**执行层**。

```
┌─────────────────────────────────────────────────────────────┐
│  应用层：LangGraph / AutoGen / CrewAI（编写 Agent 逻辑）     │
├─────────────────────────────────────────────────────────────┤
│  执行层：Agent Runtime（Aetheris）                          │
│  ├─ 任务编排与调度                                          │
│  ├─ 状态持久化与恢复                                        │
│  ├─ 外部副作用的 At-most-once 保证                          │
│  ├─ 人工介入（暂停、审批、信号）                            │
│  ├─ 审计与回放                                              │
├─────────────────────────────────────────────────────────────┤
│  基础设施：PostgreSQL / Redis / Docker                      │
└─────────────────────────────────────────────────────────────┘
```

它的核心职责是：**把 Agent 从「脆弱的脚本」变成「可靠的进程」。**

## 4. Agent Runtime 解决的核心问题

### 4.1 持久化与恢复（Durable Execution）

Agent 可能运行几分钟、几小时、甚至几天。

```
Case 1: 审批流
Day 1: Agent 生成合同 → 暂停等待审批
Day 3: 审批通过 → Agent 继续执行 → 发送合同
       ↑ 这 3 天里，Agent 状态必须完整保留

Case 2: 长任务
Hour 1: 拉取 10 万条数据
Hour 2: 数据清洗
Hour 3: 生成报告
       ↑ 每个小时都可能崩溃，必须能从断点恢复
```

**Aetheris 的方案**：事件溯源 + Checkpoint

- 每一步执行都写入**事件流**（Event Stream）
- 每一步完成后写入**检查点**（Checkpoint）
- 崩溃恢复时：从事件流重建状态，从 Checkpoint 恢复执行位置

### 4.2 At-Most-Once 语义（不重复执行外部副作用）

这是最关键的问题：**如何保证外部 API 调用不重复？**

```
退款场景：
1. Agent 调用支付 API 退款 $100
2. 支付 API 返回成功
3. Agent 准备写日志... 
4. 崩溃了！

重启后：
- 如果全部重来：用户被扣款 $200！（因为支付 API 被调用了两次）
- 如果不重来：不知道退款是否真的成功了
```

**Aetheris 的方案**：Tool Ledger（工具账本）

- 每次工具调用都生成唯一的 **Idempotency Key**
- 调用前检查 Ledger：如果已存在，直接返回上次结果
- 调用后记录结果到 Ledger

### 4.3 人工介入（Human-in-the-Loop）

很多场景需要人类批准后才能继续：

```
法务审批合同 → 需要 3 天时间等待法务确认
财务审批付款 → 需要财务点击"批准"按钮
客服升级处理 → 需要人工介入判断
```

**Aetheris 的方案**：Wait 节点 + Signal

- Agent 执行到 Wait 节点时进入 `Parked` 状态
- 等待期间不占用任何调度资源
- 外部通过 Signal（信号）唤醒 Agent 继续执行

### 4.4 审计与可调试性

生产环境的 Agent 必须能回答：

> "为什么 Agent 在第 3 步做出了这个决策？"
> "是哪次 LLM 调用导致了这个问题？"
> "Agent 调用支付 API 时的完整输入输出是什么？"

**Aetheris 的方案**：证据图（Evidence Graph）

- 每一步的输入、输出、使用的 RAG 文档、调用的工具、LLM 的响应
- 全部记录在事件流中
- 支持完整回放（Replay）用于调试

## 5. 业界类似的实践

| 产品 | 定位 | 类比 |
|------|------|------|
| **Temporal** | 工作流运行时 | Kubernetes for workflows |
| **Aetheris** | Agent 运行时 | Temporal for Agents |
| **AWS Step Functions** | 工作流编排 | 托管版 Temporal |
| **Azure Durable Functions** | 持久函数 | 微软版 Temporal |

Temporal 解决了**工作流**的可靠执行问题，Aetheris 解决的是 **Agent** 的可靠执行问题。

## 6. 什么时候需要 Agent Runtime？

**需要**：
- Agent 运行时间 > 1 分钟
- 有外部 API 调用（支付、邮件、CRM）
- 需要崩溃恢复
- 需要人工审批
- 需要审计追溯

**不需要**：
- 纯聊天（单轮问答）
- 原型/Demo
- 无副作用的纯推理任务

## 7. 结论

当 AI Agent 从「玩具」变成「生产系统」时，我们不能继续用「脚本」的思维来运行它。

**脚本是一次性的，进程是持久的。**
**脚本崩溃了重头来过，进程崩溃了从检查点恢复。**
**脚本不保证副作用，进程保证 At-most-once。**

Agent Runtime 就是给 AI Agent 配备的「操作系统」—— 它让 Agent 从脆弱的脚本，变成可靠的进程。

LLMs made agents possible.
Reliable runtimes will make agents usable in production.
